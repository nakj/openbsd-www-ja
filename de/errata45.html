Detailliertere Informationen darüber, wie man Patches unter OpenBSD installiert, findest du in der OpenBSD-FAQ.

  • 011: RELIABILITY FIX: 29. Januar 2010   alle Architekturen
    Bei Anwendung von ptrace(2) auf einen Vorläuferprozess kann innerhalb des Prozessbaums eine Schleife erzeugt werden, die gegen Annahmen in anderen Kernelteilen widerspricht und zu Endlosscheifen führen könnte.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 010: SECURITY FIX: 26. November 2009   alle Architekturen
    Das SSL/TLS-Protokoll ist anfällig für einen Man-in-the-middle-Angriff bezüglich Neuverhandlungen (siehe CVE-2009-3555, draft-ietf-tls-renegotiation-00). OpenSSL erlaubt diese Protokollfunktion standardmäßig und bietet keine Möglichkeit, sie abzuschalten.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 009: RELIABILITY FIX: 28. Oktober 2009   alle Architekturen
    getsockopt(2) mit IP_AUTH_LEVEL, IP_ESP_TRANS_LEVEL, IP_ESP_NETWORK_LEVEL der IP_IPCOMP_LEVEL wird zu einem Systemabsturz führen.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 008: RELIABILITY FIX: 5. Oktober 2009   nur i386
    XMM-Exceptions werden nicht richtig verarbeitet, was zu einem Systemabsturz führt.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 014: RELIABILITY FIX: 29. Juli 2009   alle Architekturen
    Eine Verwundbarkeit wurde in BINDs Nameserver gefunden (CVE-2009-0696). Ein Angreifer kann einen Server abstürzen lassen, indem speziell erstellte dynamische Update-Nachrichten an eine Zone gesendet werden, für die der Server der Master ist.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 006: RELIABILITY FIX: 24. Juni 2009   alle Architekturen
    Ein »off-by-one«-Fehler in der Extraktionsfunktion in Zlib.xs des Compress::Raw::Zlib-Perlmoduls vor 2.017 (CVE-2009-1391) - z. B. in AMaViS, SpamAssassin und möglicherweise anderen Produkten eingesetzt - ermöglicht es kontextabhängig Angreifern, einen Denial of Service (Hängenbleiben oder Absturz) über erzeugte zlib-Kompressionsströme hervorzurufen, da dieser Fehler zu einem heapbasierten Puffüberlauf führt.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 005: RELIABILITY FIX: 24. April 2009   alle Architekturen
    Unter sehr hoher Systemlast könnte ein Audiointerrupt eintreten, während der Audioprozess den Audio-Ringpuffer füllt. Dies löst die Verarbeitung von unsinngem (und unnötigem) Korrekturcode im audio(4)-Treiber aus, wodurch Audioapplikationen aus dem Takt geraten und somit ständiges Stottern auftritt, bis die Applikation neugestartet wird.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 004: RELIABILITY FIX: 24. April 2009   alle Architekturen
    Wenn sich aucat(1) im Servermodus und im Vollduplexmodus (Standard) befindet, wird jede Synchronisationsnachricht zweimal gesendet, wodurch Clientanwendungen davon ausgehen, dass Pufferunterläufe stattfinden. Jen ach Applikation kann dies dazu führen, dass der Sound stottert.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 003: RELIABILITY FIX: 24. April 2009   nur i386
    When DMA-fähiger Speicher von Gerätetreibern gemappt wird, sind die Mappingoptionen und der Mappingschutz teilweise uninitialisiert. Je nach Aufrufsort kann dies dazu führen, dass sich Geräte fehlerhaft verhalten, zum Beispiel dass audio(4) stottert - weitere Anomalien könnten auch mit anderen Gerätetypen beobachtet werden.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 002: RELIABILITY FIX: 11. April 2009   alle Architekturen
    Wenn pf eine Übersetzung eines gezielt angelegten IP-Datagramms vornimmt, kann eine Nullpointer-Dereferenzierung eintreten, die zu einer Kernelpanic führt. In bestimmten Konfigurationen kann dies von einem Netzwerkangreifer ausgelöst werden.
    Mit der Begrenzung von Übersetzungsregeln auf Protokolle, die für die genutze IP-Version spezifisch sind, steht eine effektive Möglichkeit zur Umgehung des Problems bereit, bis der Patch installiert werden kann. Für IPv4-nat/binat/rdr-Regeln kann beispielsweise Folgendes genutzt werden: 
        nat/rdr ... inet proto { tcp udp icmp } ...
    Oder für IPv6-nat/binat/rdr-Regeln kann Folgendes genutzt werden: 
        nat/rdr ... inet6 proto { tcp udp icmp6 } ...
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 001: RELIABILITY FIX: 8. April 2009   alle Architekturen
    Der Verarbeitungscode von OpenSSL für ASN.1 kann über bestimmte ungültige Zeichenketten (CVE-2009-0590) oder unter bestimmten Fehlerbedingungen, die über ungültige ASN.1-Strukturen erreicht werden (CVE-2009-0789), ungültige Speicherzugriffe durchführen. Diese Verwundbarkeiten könnten ausgenutzt werden, um einen Denial-of-Service-Angriff durchzuführen. Eine ausführlichere Beschreibung dieser Probleme befindet sich im OpenSSL security advisory; beachte jedoch, dass die anderen aufgelisteten Probleme unter "Incorrect Error Checking During CMS verification" sich auf Code bezieht, der unter OpenBSD nicht aktiviert wurde.
    Es existiert ein Quelltextpatch, der das Problem behebt.
